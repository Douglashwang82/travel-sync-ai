# TravelSync AI 系統文件

> **專案狀態**：MVP 已實作
> **主要導向**：LINE 旅遊規劃 AI 助手，整合群組對話解析與智慧決策。

---

## 1. 系統概述 (System Overview)

TravelSync AI 是一個嵌入在 LINE 群組中的旅遊規劃助手。它透過 LLM (Gemini 1.5 Flash/Pro) 解析群組成員的自然語言對話，自動提取旅遊相關的實體資訊（如日期、地點），並轉化為結構化的旅遊代辦清單。系統提供視覺化的投票介面，協助群組成員在 LINE 內完成飯店、餐廳等決策。

### 技術棧 (Tech Stack)
- **前端 (LIFF)**: Next.js + Tailwind CSS + Shadcn UI
- **後端 (API/Sync)**: Next.js API Routes (Serverless)
- **基礎設施**: Vercel (Hosting + Cron Jobs)
- **資料庫/認證**: Supabase (PostgreSQL + RLS + Admin Client)
- **AI 引擎**: Google Gemini API (1.5 Flash)
- **外部整合**: LINE Messaging API, Google Places API
- **監控/分析**: Sentry (Error Tracking), Posthog (Analytics)

## 2. 核心架構與檔案結構 (Architecture & Directory)

專案採用 Next.js App Router 結構，邏輯拆分為五大塊：

### 📁 檔案目錄說明
- `/app`: 系統入口、Webhook、CRON 路由與 LIFF 頁面。
- `/bot`: LINE 指令解析器與訊息路由邏輯。
- `/services`: 核心業務邏輯，包含 AI 解析、投票決策、行程狀態管理。
- `/lib`: 共用工具，包含 DB 連線、LINE SDK 封裝、分析追蹤。
- `/supabase`: 資料庫遷移檔案 (Migrations) 與 Schema 定義。
- `/components`: LIFF 專用的 UI 元件。
- `.agents/rules`: AI 輔助開發的規範文件。

## 3. 主要功能模組 (Feature Modules)

### 3.1 智慧對話解析 (AI Parsing Engine)
- **入口**: `/app/api/line/webhook` 接收事件，轉交 `services/parsing` 處理。
- **流程**: 
  1. **Relevance Filter**: 基於規則過濾無關訊息。
  2. **Context Assembly**: 抓取群組內目前的行程背景。
  3. **AI Extraction**: 使用 Gemini 提取 `hotel`, `activity`, `date_range` 等實體。
  4. **Item Generation**: 自動在看板建立 `todo` 項目。
  5. **Conflict Detection**: 偵測日期衝突，並標記為 `pending` 待決策。

### 3.2 決策與投票系統 (Decision & Vote)
- **入口**: 指令 `/vote [item]` 或 AI 偵測到衝突時觸發。
- **流程**:
  1. `services/decisions`: 串接 Google Places API 搜尋候選地點。
  2. 生成 **LINE Flex Message** 投票卡片。
  3. 成員於 LINE 內投票，數據存儲於 `votes` 表。
  4. **Cron Job**: `/api/cron/vote-deadlines` 定時檢查並公布贏家。

### 3.3 行程看板 (LIFF Dashboard)
- **URL**: `/liff/dashboard`
- **功能**: 展示 **Todo**、**Pending** (投票中)、**Confirmed** (已確認) 三階段看板。
- **實作**: 透過 LIFF SDK 取得 LINE User Profile，並與 Supabase 進行後端驗證。

## 4. 資料模型 (Data Model)

系統基於 PostgreSQL (Supabase) 運行，核心資料表包含：
- `line_groups`: 儲存 LINE 群組與狀態。
- `trips`: 每個群組對應的一個活躍行程 (MVP 限定一個)。
- `trip_items`: 看板上的具體條目 (飯店、餐廳、保險等)。
- `trip_item_options`: 投票選項 (連結至 Google Place ID)。
- `votes`: 使用者投票紀錄。
- `line_events`: 完整紀錄 Webhook 收到的一律事件（具備 `status` 與 `retry_count`）。
- `parsed_entities`: AI 提取出的原始實體紀錄與信心分數。

## 5. 環境變數設定 (.env)

專案需要以下環境變數才能運作：

### LINE 平台
- `LINE_CHANNEL_ACCESS_TOKEN`
- `LINE_CHANNEL_SECRET`
- `NEXT_PUBLIC_LIFF_ID`

### Supabase
- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY` (用於後台操作)

### AI & API
- `GEMINI_API_KEY`
- `GOOGLE_MAPS_API_KEY` (用於 Places 搜尋)

### 指令集
- `CRON_SECRET`: 用於驗證 Vercel Cron Job 請求。

## 6. 指令清單 (LINE Commands)

| 指令 | 說明 |
| :--- | :--- |
| `/start [目的地] [日期]` | 初始化一個新的行程。 |
| `/status` | 顯示目前的看板進度摘要。 |
| `/vote [項目名稱]` | 針對特定待辦事項啟動投票模式。 |
| `/nudge` | 提醒尚未投票的成員。 |
| `/add [項目]` | 手動新增待辦事項。 |
| `/optout` | 成員選擇不參與 AI 訊息分析。 |
| `/help` | 顯示所有功能說明。 |

## 7. 部署說明 (Deployment)

1. **資料庫**: 執行 `supabase/migrations` 下所有 SQL。
2. **LINE OA**: 
   - 設定 Webhook URL 為 `https://YOUR_DOMAIN/api/line/webhook`。
   - 啟用 LIFF 並設定跳轉 URL。
3. **Vercel**: 
   - 部署 Next.js 應用。
   - 設定 `vercel.json` 中的 Cron 配置。
4. **Rich Menu**: 執行 `scripts/setup-rich-menu.ts` 建立 LINE 底部的專頁選單。
